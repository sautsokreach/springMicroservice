Step-by-Step Auth Flow
1
Client → Keycloak: Request Token
Browser: GET /auth/realms/microservice-realm/protocol/openid-connect/auth?client_id=gateway&response_type=code&scope=openid+profile&code_challenge=…&code_challenge_method=S256
Service-to-service: POST /protocol/openid-connect/token with grant_type=client_credentials
2
Keycloak: Authenticate + Issue JWT
Keycloak validates credentials, maps roles from realm / client scopes, and returns a signed JWT (RS256).
Token contains: sub, realm_access.roles, resource_access, exp, iat
3
Client → Gateway: API Request with Bearer Token
Authorization: Bearer <access_token> is included in every API request sent to the Gateway.
4
Gateway: JWT Validation via JWKS
Spring Security Resource Server filter fetches JWKS from http://keycloak:8180/realms/microservice-realm/protocol/openid-connect/certs, verifies JWT signature, expiry and audience. Rejected requests get 401 Unauthorized immediately — services are never called.
5
Gateway → Downstream Services
Gateway strips the raw JWT and propagates only decoded user headers: X-Auth-User-Id, X-Auth-User-Roles, X-Auth-User-Email. Each downstream service also acts as a resource server and re-validates with @PreAuthorize("hasRole('USER')").
6
Token Refresh
When access token expires, client sends refresh token to POST /protocol/openid-connect/token with grant_type=refresh_token. Keycloak issues a new access token without requiring re-authentication.

Messaging Flow: Order Placement (Event-Driven)
#ComponentActionBrokerMessage
1 API Gateway Routes authenticated POST /api/v1/orders to Order Service — HTTP request with JWT claims
2 Order Service Validates request, persists Order (status=PENDING) in order-db — SQL INSERT
3 Order Service → Kafka Publishes ORDER_PLACED event via Spring Cloud Stream Kafka order-events { orderId, items[], userId }
4 Product Service Consumes ORDER_PLACED, decrements stock for each item Kafka order-events Inventory adjustment
5 Product Service → Kafka Publishes INVENTORY_UPDATED event Kafka product-inventory { productId, newStock }
6 Order Service Consumes INVENTORY_UPDATED, updates Order status=CONFIRMED Kafka product-inventory Status update
7 Order Service → RabbitMQ Publishes ORDER_CONFIRMED notification via Spring Cloud Stream RabbitMQ order.notifications { orderId, userId, status: CONFIRMED }
8 User Service Consumes notification, sends email/push to user RabbitMQ order.notifications Notification dispatch
POST /api/v1/orders — Full Authenticated Request Flow
#FromToDetail
1 Client Keycloak :8180 POST /token with credentials → receive JWT access token (RS256)
2 Client Ingress :443 HTTPS POST /api/v1/orders with Authorization: Bearer <JWT>
3 Ingress Gateway :8080 Forward to gateway pod (TLS terminated)
4 Gateway Keycloak JWKS Validate JWT signature · Check expiry · Extract roles
→ 401 if invalid, else continue
5 Gateway Eureka :8761 Resolve lb://order-service → get healthy pod IPs
6 Gateway Order Service :8082 Forward request + X-Auth-User-* headers + traceId (Zipkin B3)
7 Order Service User Service :8081 Verify user exists via Circuit Breaker @LoadBalanced call
CB: CLOSED → proceed; CB: OPEN → fallback
8 Order Service Product Service :8083 Validate product exists + price via Circuit Breaker call
Retry max 3× with 1s backoff · Timeout 3s
9 Order Service order-db INSERT order record (status=PENDING) → PostgreSQL via JPA
10 Order Service Kafka: order-events Publish ORDER_PLACED event via Spring Cloud Stream Kafka binder
11 Order Service Gateway → Client Return 201 Created with traceId in response headers
12 All services Zipkin :9411 Async span export — full trace visible in Zipkin UI with all hops

Requests pass through
Tracking failure rate

STATE

Fallback invoked

Success → CLOSED
Fail → OPEN
ParameterValueEffect
slidingWindowSize10Track last 10 calls for failure rate calculation
failureRateThreshold50%Open circuit when ≥50% of calls fail
waitDurationInOpenState5sStay OPEN for 5s before attempting HALF-OPEN
permittedCallsInHalfOpen3Allow 3 probe calls in HALF-OPEN state
Retry maxAttempts3Retry up to 3 times before counting as failure
Retry waitDuration1sWait 1 second between retry attempts
TimeLimiter timeout3sAbort call if it takes longer than 3 seconds
Annotated Fallback Flow
#EventComponentResult
1 Order Service calls User Service OrderServiceClient @CircuitBreaker(name="user-service", fallbackMethod="getUserFallback")
2 Call times out after 3s or throws exception TimeLimiter / CB Retry #1 → wait 1s → Retry #2 → wait 1s → Retry #3
3 All 3 retries fail CircuitBreaker Failure recorded in sliding window; if rate ≥ 50% → OPEN
4 Circuit is OPEN getUserFallback() Throws ServiceUnavailableException → 503 to caller
5 After 5s waitDuration CircuitBreaker Transitions to HALF-OPEN; 3 probe calls allowed
6 Probe calls succeed CircuitBreaker Transitions back to CLOSED; normal operation resumes
